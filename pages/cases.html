Backoffice Requests

Что было: Обработка заявок выполнялась вручную через корпоративную почту. Не было единой платформы для исключения дубликатов, сотрудники не имели точной информации о текущем статусе заявки. Адреса указывались по-разному, отсутствовало разделение по типам задач, все проверки (включая согласование внутреннего контроля) выполнялись вручную.

Что сделал:

  • Разработал и внедрил workflow для всех типов заявок (QR/POS, E-Com, расторжение договоров, тарифы, реквизиты и др.).
  • Настроил динамические формы (Extension for Jira Service Desk).
  • Создал Groovy-скрипты для:
    • автосоздания связанных задач в проектах GTE, KKM, BANKPOS, DCE,
    • копирования определённого вложения (среди 10+ файлов) в другой проект через ScriptRunner-Jobs, с валидацией по наименованию,
    • назначения согласующих и автоматической отметки “approved”
  • Добавил валидации для:
    • проверка ИНН,
    • проверка адреса (поле мультивыбор)
    • проверка согласующих лиц
  • Настроил кастомные поля, схемы типов задач и схемы экранов.
  • Настроил экраны переходов, где при смене статуса открываются обязательные для заполнения поля.
    • Реализовал в JIRA Service Desk панель «Заявки по этому ИНН» для быстрого поиска и отображения всех связанных задач.

Результат:

  • Устранено дублирование заявок
  • Повысилась прозрачность процессов (видно текущий статус).
  • Автоматизированы проверки и согласование.
  • Сокращено время обработки заявок и снижено количество ошибок.
  • Скорость обработки заявок выросла в 3 раза, количество ошибок снизилось на 40%.

Telegram Bot (Jira Integration)

Что было: Отчёты формировались вручную, поиск заявок занимал много времени. Полевым сотрудникам было неудобно заходить через телефон в Jira и проверять свои заявки.

Что сделал:

  • Подключил бота к Jira API.
  • Реализовал автоматическую рассылку отчётов: ежедневные, еженедельные, ежемесячные.
  • Интегрировал бота с FastAPI DBC System (Data Base Control System) для хранения данных пользователей и статистики.
  • Добавил двухэтапную аутентификацию:
    • /start — проверка корпоративной почты и пароля через базу данных, доступ только авторизованным пользователям.
  • Реализовал быстрый поиск заявок:
    • /where — поиск в проекте BO (ИНН, статус, номер заявки, юр. название, автор, исполнитель).
    • /where_buh — поиск в проекте GTE для бухгалтерии (тема и ссылка на портал Service Desk).
    • /where_lk — выдача логина и пароля от личного кабинета партнёра.
  • Поиск ведётся только по юридическому наименованию, что обеспечивает точность.
  • Сделал рассылку Excel-отчётов прямо в Telegram.
  • Поднял систему сбора статистики по заявкам (по ИНН, по проектам, по времени обработки) через связку Bot + FastAPI + Jira.

Результат:

  • Отчёты формируются и доставляются автоматически, без ручного участия.
  • Поиск заявок занимает секунды вместо часов.
  • Бухгалтерия получает нужные заявки в несколько кликов.
  • Полевые сотрудники могут оперативно проверять заявки через Telegram, не заходя в Jira.
  • В критически важных ситуациях стало возможным мгновенно получить данные из Jira или ЛК.
  • Собранная статистика используется для анализа эффективности Backoffice и улучшения KPI.
  • Увеличена безопасность: доступ к данным возможен только после аутентификации.

  • Подключил к платформе Telegram Bot для централизованной обработки команд.
  • Реализовал сбор статистики по заявкам и активности пользователей.

Результат:

  • Создан единый центр управления интеграциями (Jira ↔ Bot).
  • Система стала масштабируемой и удобной для расширения.
  • Все пользовательские данные хранятся централизованно и безопасно.
